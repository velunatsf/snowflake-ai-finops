Spend Controls

New SERVICE_TYPE billing, per-user credit limits, and Cortex Code cost governance

Learning Objectives

  • Detect the 5 SERVICE_TYPE entries that track AI spend in METERING_DAILY_HISTORY
  • Set per-user credit limits and model allowlists to control Cortex Code spend
  • Apply the four-layer governance model: Credit Limits → Budgets → Model Allowlist → Usage History
  • Query CORTEX_CODE_CLI_USAGE_HISTORY for per-user, per-model attribution
  • Use AI_COUNT_TOKENS to estimate cost before execution and TRY_COMPLETE for production safety
A

New SERVICE_TYPE Billing New Feature

Breaking Change for Cost Dashboards

Four services are getting their own billing types as part of the AI Credits transition. If your cost dashboards only look for AI_SERVICES, you will silently miss spend from these new service types.

Moving to Own SERVICE_TYPE

  • CORTEX_AGENTS
  • CORTEX_CODE_CLI
  • CORTEX_CODE_SNOWSIGHT
  • SNOWFLAKE_INTELLIGENCE

Still Under AI_SERVICES

  • AI Functions (COMPLETE, SENTIMENT, etc.)
  • Cortex Analyst
  • Fine Tuning
  • Search Serving
  • Provisioned Throughput

Query: Detect New AI Credit Service Types

Transition Check

When to run: Immediately - check if your account already shows the new service types in METERING_DAILY_HISTORY.

SQL 
SELECT
    SERVICE_TYPE,
    MIN(USAGE_DATE)          AS first_seen,
    MAX(USAGE_DATE)          AS last_seen,
    SUM(CREDITS_BILLED)      AS total_credits_billed,
    SUM(CREDITS_USED)         AS total_credits_used
FROM SNOWFLAKE.ACCOUNT_USAGE.METERING_DAILY_HISTORY
WHERE SERVICE_TYPE IN (
    'AI_SERVICES',
    'CORTEX_AGENTS',
    'CORTEX_CODE_CLI',
    'CORTEX_CODE_SNOWSIGHT',
    'SNOWFLAKE_INTELLIGENCE'
)
GROUP BY SERVICE_TYPE
ORDER BY total_credits_billed DESC;
Action Required for Your Dashboards

If you built cost dashboards that filter on SERVICE_TYPE = 'AI_SERVICES', update them now to also include the four new service types. Otherwise, your AI spend totals will be silently wrong - showing lower costs than actual.

B

Spend Cap - Cortex Code Credit Guardrails New Feature

Spend Caps on Budgets

Enable financial control by setting automated spend limits on your cloud expenses. Uses out-of-the-box budgets to enforce automated cost boundaries at the project level for specific services in a non-destructive manner. On budget breach, usage is blocked and financial damage is controlled - reduce bill shock, and protect your Cortex budget!

How do you govern LLM spend from Cortex Code on a per-user basis? Four tools, each for a different scenario:

Layer Tool Behavior Scope
Proactive Per-user credit limits Hard block - rolling 24h window Account-wide default + per-user override
Threshold Monthly Budgets Alert before overshoot - no hard block Account-wide
Preventive Model Allowlist Block expensive models at the source Account-wide
Audit Usage History views Attribution after-the-fact by user + model Per surface (CLI / Snowsight)

Per-User Credit Limits Proactive - Blocks Before Spend Happens

Parameter Controls
CORTEX_CODE_CLI_DAILY_EST_CREDIT_LIMIT_PER_USER Snowflake CoCo CLI usage
CORTEX_CODE_SNOWSIGHT_DAILY_EST_CREDIT_LIMIT_PER_USER Cortex Code in Snowsight usage

Set Account-Wide Default + Per-User Override

Guardrail SQL
SQL 
-- Account-wide default: 20 credits/day per user (CLI)
ALTER ACCOUNT SET CORTEX_CODE_CLI_DAILY_EST_CREDIT_LIMIT_PER_USER = 20;

-- Tighter limit for a specific user
ALTER USER USER_COCO SET CORTEX_CODE_CLI_DAILY_EST_CREDIT_LIMIT_PER_USER = 5;

-- Same for Snowsight
ALTER ACCOUNT SET CORTEX_CODE_SNOWSIGHT_DAILY_EST_CREDIT_LIMIT_PER_USER = 20;
ALTER USER USER_COCO SET CORTEX_CODE_SNOWSIGHT_DAILY_EST_CREDIT_LIMIT_PER_USER = 5;

Rolling 24h Window

The limit is a rolling 24-hour window, not a calendar day. If a user hits the cap at 9am Tuesday, they're blocked until 9am Wednesday.

Independent Surfaces

CLI and Snowsight are tracked independently with separate parameters. Cap someone on CLI while leaving Snowsight open, or vice versa.

Two-Level Control

Set an account-wide default, then override per user. Power users get more headroom; restricted roles get tighter limits.

Monthly Budgets Threshold - Alerts Before Overshoot

Snowflake Budgets track actual billed credits across warehouses, pipelines, and AI together. Set a monthly limit and get notified before you hit it.

Budgets Alert - They Don't Hard-Block

Budgets are an early warning system. Per-user credit limits are the only hard stop for Cortex Code spend.

Model Allowlist Preventive - Block Expensive Models

Restrict which LLM models users can call via Cortex AI functions. By limiting the allowlist to budget/standard models, you prevent expensive premium model usage at the source - before any credits are consumed.

Setting Behavior
ALL (default) All available models can be used - no restriction
NONE Blocks all Cortex LLM calls entirely
Comma-separated list Only listed models are permitted
SQL 
-- Check current allowlist
SHOW PARAMETERS LIKE 'CORTEX_MODELS_ALLOWLIST' IN ACCOUNT;

-- Restrict to budget and standard models only (block premium)
ALTER ACCOUNT SET CORTEX_MODELS_ALLOWLIST = 'mistral-7b,llama3.1-8b,llama3.1-70b,mistral-large2,snowflake-arctic';

-- Allow all models (default)
ALTER ACCOUNT SET CORTEX_MODELS_ALLOWLIST = 'ALL';
Layered Defense

Combine with per-user credit limits for maximum control: the allowlist prevents expensive model selection, while credit limits cap total spend regardless of model choice.

Cortex Code Usage History Audit - Attribution After-the-Fact

CORTEX_CODE_CLI_USAGE_HISTORY

Breaks down Snowflake CoCo CLI spend by user and model. One row per usage event. Use this to identify who's spending the most and on which models.

CORTEX_CODE_SNOWSIGHT_USAGE_HISTORY

Same structure for Snowsight-based Cortex Code usage. CLI and Snowsight tracked in separate views - query both for complete attribution.

Query: Top Cortex Code Spenders by User & Model

Attribution
SQL 
-- Snowflake CoCo CLI spend by user and model (last 30 days)
SELECT
    h.USER_NAME,
    f.key                       AS model_name,
    SUM(h.TOKEN_CREDITS)        AS total_credits,
    ROUND(total_credits * 3, 4) AS est_cost_usd
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_CODE_CLI_USAGE_HISTORY h,
     LATERAL FLATTEN(input => h.TOKENS_GRANULAR) f
WHERE h.USAGE_TIME >= DATEADD('day', -30, CURRENT_TIMESTAMP())
GROUP BY h.USER_NAME, f.key
ORDER BY total_credits DESC;

-- Repeat with CORTEX_CODE_SNOWSIGHT_USAGE_HISTORY
-- for Snowsight usage attribution

Resource Monitors Do NOT Cover AI Token Costs

Resource Monitors only cap warehouse compute credits. If that's your current AI cost control strategy, you have a gap. Use per-user credit limits as the hard stop, Budgets for early warning, and the usage history views for attribution.

C

Helper Functions for Cost Control

Two helper functions let you estimate cost before execution and handle failures gracefully at scale:

AI_COUNT_TOKENS

Estimate token consumption before running a query. Understand how prompt context impacts cost without spending any AI credits - only compute cost applies.

  • Estimate cost before scaling to full dataset
  • Compare token counts across prompt designs
  • Validate that input fits within model context window
  • Works with AI_COMPLETE, AI_SENTIMENT, AI_CLASSIFY, AI_EMBED, AI_TRANSLATE, AI_SIMILARITY

TRY_COMPLETE

Run AI_COMPLETE but return NULL instead of an error when input exceeds model limits. Prevents query failures when processing variable-length text at scale.

  • Prevents batch query failures on oversized rows
  • Returns NULL for rows that exceed context window
  • Process millions of rows without one bad row killing the job
  • Use with SNOWFLAKE.CORTEX.TRY_COMPLETE()

AI_COUNT_TOKENS - Estimate Before You Spend

Cost Estimation
SQL 
-- Count tokens for an AI_COMPLETE call (no AI credits consumed)
SELECT AI_COUNT_TOKENS('ai_complete', 'mistral-7b',
    'Summarize this feedback: The product arrived late and was damaged.');

-- Compare token counts across models for the same prompt
SELECT
    'mistral-7b' AS model,
    AI_COUNT_TOKENS('ai_complete', 'mistral-7b', prompt) AS tokens
FROM (SELECT 'Summarize: ' || c_comment AS prompt
      FROM SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.CUSTOMER LIMIT 1)
UNION ALL
SELECT
    'llama3.1-70b',
    AI_COUNT_TOKENS('ai_complete', 'llama3.1-70b', prompt)
FROM (SELECT 'Summarize: ' || c_comment AS prompt
      FROM SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.CUSTOMER LIMIT 1);

-- Estimate total tokens before running at scale
SELECT
    COUNT(*) AS total_rows,
    SUM(AI_COUNT_TOKENS('ai_complete', 'mistral-7b',
        CONCAT('Summarize: ', c_comment))) AS total_input_tokens,
    ROUND(total_input_tokens / 8300000, 6) AS est_credits
FROM SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.CUSTOMER;

TRY_COMPLETE - Graceful Failure at Scale

Production Safety
SQL 
-- TRY_COMPLETE returns NULL instead of failing on oversized input
SELECT
    customer_id,
    SNOWFLAKE.CORTEX.TRY_COMPLETE('mistral-7b',
        CONCAT('Summarize: ', feedback_text)
    ) AS summary
FROM cortex_lab.ai_workshop.customer_feedback;

-- Count how many rows would fail (NULL = exceeded context window)
SELECT
    COUNT(*) AS total_rows,
    COUNT(SNOWFLAKE.CORTEX.TRY_COMPLETE('mistral-7b',
        CONCAT('Summarize: ', feedback_text))) AS successful_rows,
    total_rows - successful_rows AS failed_rows
FROM cortex_lab.ai_workshop.customer_feedback;
FinOps Pattern: Estimate, Then Execute

Use AI_COUNT_TOKENS to estimate cost before running at scale, then use TRY_COMPLETE in production to handle edge cases gracefully. This prevents both bill shock and batch failures.
