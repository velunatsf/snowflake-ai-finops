AI Token Economy

Understanding the fundamental unit of AI cost in Snowflake

Learning Objectives

  • Understand what a token is and how it maps to cost
  • Know the two cost layers in Snowflake AI workloads
  • Estimate query cost before running at scale
A

What Is a Token?

Every AI model processes text by breaking it into smaller pieces called tokens. These tokens are the fundamental billing unit for AI operations in Snowflake Cortex.

Token Basics

  • 1 token ≈ 0.75 English words
  • Common words = 1 token
  • Long/uncommon words = multiple tokens
  • Spaces and punctuation = tokens
  • Both input AND output count toward cost

Quick Estimates

  • 100 words ≈ 130 tokens
  • 1 page of text ≈ 500 tokens
  • 1 short email ≈ 100–200 tokens
  • 1 SQL query ≈ 50–150 tokens

Prompt + Response Anatomy

Every AI call has two token components that get billed:

PROMPT (Input Tokens)
System message : 50 tokens
Your instruction : 30 tokens
Data context : 200 tokens
RESPONSE (Output Tokens)
Model reply : 150 tokens
Total this call: 430 tokens
Key Insight

Output tokens typically cost more than input tokens for premium models. A verbose model response can cost significantly more than a concise one - this is why prompt engineering to request brief answers is a real cost optimization technique.

B

Two Cost Layers in Snowflake AI

When you run AI workloads in Snowflake, you're dealing with two distinct cost layers. Understanding this distinction is crucial for FinOps.

Layer 1: Warehouse Compute Credits

  • Traditional Snowflake billing
  • Based on warehouse size × time running
  • Stops when warehouse suspends
  • You control this via auto-suspend settings
Example: SMALL warehouse = 2 credits/hour

Layer 2: AI Token Credits (Cortex)

  • New billing layer for AI functions
  • Based on tokens consumed per call
  • Does NOT stop when warehouse suspends
  • Some functions run serverless
Example: 1M tokens on claude-4-sonnet = 1.5-7.5 credits
Critical Cost Warning

Token costs do not stop when you suspend your warehouse. A misconfigured loop calling AI_COMPLETE() across 1 million rows can exceed a week of warehouse compute in minutes. This is the #1 surprise cost for teams adopting Cortex AI.

Serverless vs. Warehouse-Based

Execution Mode Functions Cost Behavior
Serverless AI_COMPLETE, AI_SENTIMENT, AI_SUMMARIZE_AGG, AI_TRANSLATE, AI_CLASSIFY Token credits only - no warehouse needed
Warehouse + Serverless Cortex Search, Cortex Analyst Both warehouse compute AND token credits
Warehouse-based FORECAST, ANOMALY_DETECTION Primarily warehouse compute

Optimized Internal Models (Billed Per Character)

These functions use Snowflake's own optimized internal models - not LLMs like mistral or claude. They process raw text directly by character count, not through a tokenizer. Snowflake bills them at a flat rate per character processed.

Shadow Waste Savings: Task-Specific Functions vs AI_COMPLETE

The most common shadow waste pattern: using AI_COMPLETE() for tasks that have a dedicated, cheaper function. The table below shows expected savings when switching - validate with your own workload before projecting.

Instead of… Use… Expected Savings Why
AI_COMPLETE('Classify sentiment…') AI_SENTIMENT(text) 40–70% Purpose-built model, no prompt tokens, lower per-call cost
AI_COMPLETE('Classify into: A, B, C…') AI_CLASSIFY(text, […]) 50–75% Zero-shot classification without prompt engineering overhead
AI_COMPLETE('Translate to French…') AI_TRANSLATE(text, 'fr') 40–60% Optimized translation model, 90+ languages, no prompt tokens
AI_COMPLETE('Summarize this…') with premium model AI_COMPLETE() with mistral-7b 30–60% Budget model usually sufficient for summarization
AI_COMPLETE() with claude-4-sonnet AI_COMPLETE() with mistral-7b 10–50x Right-sizing: cheapest model that meets quality bar

Standing Cost: Cortex Search Services

Unlike pay-per-call AI functions, Cortex Search has a standing cost model with two cost components that accrue regardless of query volume:

Cost Component Rate Trigger
Serving Compute 6.3 credits/GB/month Continuous while service is resumed, even with zero queries
Indexing (Warehouse + AI_EMBED tokens) Variable On source data change: warehouse refreshes + token embedding costs

Key distinction: Serving credits burn 24/7 as long as the service status is "resumed." Indexing costs fire only when source data changes, but for active tables, that can be continuous too.

  • An idle Search service with serving resumed = 100% waste on serving credits
  • Weekly audit: is each service actively queried?
  • Use CORTEX_SEARCH_DAILY_USAGE_HISTORY with CONSUMPTION_TYPE = 'SERVING' to find silent cost
  • Suspend unused services or DROP them immediately
  • See Module 06, Query 6: Idle Cortex Search Detection for the full detection query
SQL 
-- Suspend serving only (stops the 6.3 credits/GB/month standing cost)
ALTER CORTEX SEARCH SERVICE my_search_svc SUSPEND SERVING;

-- Suspend indexing only (stops warehouse + embedding costs on data changes)
ALTER CORTEX SEARCH SERVICE my_search_svc SUSPEND INDEXING;

-- Suspend both serving AND indexing
ALTER CORTEX SEARCH SERVICE my_search_svc SUSPEND;

-- Resume when needed (typically takes a few minutes)
ALTER CORTEX SEARCH SERVICE my_search_svc RESUME;

-- Nuclear option: permanently remove an abandoned service
DROP CORTEX SEARCH SERVICE IF EXISTS my_abandoned_svc;

Knowledge Check

Q1: A 500-token prompt generates a 300-token response. How many total tokens are billed?

800 tokens total. Both input (prompt) and output (response) tokens are billed. This is why limiting response length is a cost optimization technique.

Q2: Which cost layer stops when you suspend your warehouse?

Layer 1: Warehouse Compute Credits. Layer 2 (AI Token Credits) continues to accumulate on every Cortex function call, regardless of warehouse state. Some Cortex functions run serverless and don't use a warehouse at all.

Q3: What does "serverless Cortex" mean for cost tracking?

You pay only for tokens, not warehouse time. Functions like AI_SENTIMENT(), AI_COMPLETE(), and AI_TRANSLATE() run on Snowflake's managed infrastructure. This is efficient for sporadic use, but can accumulate rapidly at scale since there's no auto-suspend safety net.
