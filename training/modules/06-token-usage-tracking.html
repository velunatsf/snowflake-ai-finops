Token & Credit Usage Tracking

Your real-time FinOps console for AI spend

Learning Objectives

  • Query live token and credit consumption
  • Attribute cost by user, function, and model
  • Set up cost alerts before problems occur
Bookmark These Queries

Everything in this module runs in Snowsight. These queries are your real-time FinOps console for AI spend. Save them as worksheets for quick access.

Latency Note

account_usage views have approximately 45 minutes of latency. Very recent queries from Module 05 may not appear immediately. For real-time monitoring, use the INFORMATION_SCHEMA alternative at the bottom of this module.

Cloud Services Credits ≠ AI Token Credits

query_history.credits_used_cloud_services measures cloud services overhead (query compilation, metadata) - not AI token credits. A Cortex call costing 0.5 AI credits may show only 0.000036 in cloud services.

The real AI cost lives in dedicated views: CORTEX_AI_FUNCTIONS_USAGE_HISTORY (CREDITS column) and CORTEX_AISQL_USAGE_HISTORY (TOKEN_CREDITS column). Queries 2–6 below use these dedicated views.

REF

AI Cost Tracking - ACCOUNT_USAGE Views

Five views in SNOWFLAKE.ACCOUNT_USAGE cover AI spend. Each has different granularity, latency (~45 min), and key columns.

Resource Monitors Do NOT Cover AI/Serverless

Resource Monitors cap warehouse compute credits only. To track and cap AI token credits, use Snowflake Budgets and the views below.

Query 1: AI Credits by Cortex Function

Weekly Review

When to run: Weekly FinOps review - uses the dedicated CORTEX_AI_FUNCTIONS_USAGE_HISTORY view for accurate AI token credits (not cloud services overhead).

SQL 
SELECT
    FUNCTION_NAME,
    COUNT(*)                                    AS call_count,
    ROUND(SUM(CREDITS), 4)                      AS total_ai_credits,
    ROUND(AVG(CREDITS), 6)                      AS avg_credits_per_call,
    ROUND(MIN(CREDITS), 6)                      AS min_credits,
    ROUND(MAX(CREDITS), 6)                      AS max_credits,
    ROUND(SUM(CREDITS) * 3, 2)                  AS est_dollars
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AI_FUNCTIONS_USAGE_HISTORY
WHERE START_TIME >= DATEADD(day, -7, CURRENT_TIMESTAMP)
GROUP BY FUNCTION_NAME
ORDER BY total_ai_credits DESC;

Query 2: Model Cost Comparison

Key Insight

When to run: After Exercise 4 in Module 05 - see real AI credit cost per model from CORTEX_AI_FUNCTIONS_USAGE_HISTORY, not estimates.

SQL 
SELECT
    MODEL_NAME,
    FUNCTION_NAME,
    COUNT(*)                                    AS calls,
    ROUND(SUM(CREDITS), 4)                      AS total_ai_credits,
    ROUND(AVG(CREDITS), 6)                      AS credits_per_call,
    ROUND(SUM(CREDITS) * 3, 2)                  AS est_dollars
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AI_FUNCTIONS_USAGE_HISTORY
WHERE START_TIME >= DATEADD(hour, -2, CURRENT_TIMESTAMP)
  AND FUNCTION_NAME IN ('COMPLETE', 'AI_COMPLETE')
GROUP BY MODEL_NAME, FUNCTION_NAME
ORDER BY total_ai_credits DESC;

Query 3: Hourly AI Spend Trend

Pattern Analysis

When to run: To identify usage patterns and anomalies - tracks real AI credits from the dedicated view, not cloud services overhead.

SQL 
SELECT
    DATE_TRUNC('hour', START_TIME)              AS hour_bucket,
    FUNCTION_NAME,
    COUNT(*)                                    AS ai_calls,
    ROUND(SUM(CREDITS), 4)                      AS ai_credits,
    ROUND(SUM(CREDITS) * 3, 2)                  AS est_dollars
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AI_FUNCTIONS_USAGE_HISTORY
WHERE START_TIME >= DATEADD(day, -1, CURRENT_TIMESTAMP)
GROUP BY 1, 2
ORDER BY 1 DESC, ai_credits DESC;

Query 4: Cost Projection Query

Non-Negotiable

When to run: BEFORE running any AI at scale. Use after Module 05 Exercise 5.

SQL 
-- Step 1: Find your recent AI call costs from the dedicated view:
SELECT
    QUERY_ID,
    FUNCTION_NAME,
    MODEL_NAME,
    ROUND(CREDITS, 6)                           AS ai_credits,
    START_TIME
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AI_FUNCTIONS_USAGE_HISTORY
WHERE START_TIME >= DATEADD(hour, -1, CURRENT_TIMESTAMP)
ORDER BY START_TIME DESC
LIMIT 10;
SQL 
-- Step 2: Project costs at scale using real AI credits:
WITH sample_cost AS (
    SELECT
        SUM(CREDITS)                            AS sample_credits,
        COUNT(*)                                AS sample_calls
    FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AI_FUNCTIONS_USAGE_HISTORY
    WHERE QUERY_ID IN (
        -- *** PASTE YOUR TEST QUERY IDs HERE ***
        'your_query_id_1',
        'your_query_id_2'
    )
)
SELECT
    sample_credits,
    sample_calls,
    ROUND(sample_credits / NULLIF(sample_calls, 0), 6) 
                                                AS credits_per_call,
    ROUND(sample_credits * 100, 4)              AS projected_500_rows,
    ROUND(sample_credits * 10000, 2)            AS projected_50k_rows,
    ROUND(sample_credits * 100000, 2)           AS projected_500k_rows,
    -- Dollar estimates at ~$3/credit
    ROUND(sample_credits * 100000 * 3, 2)       AS projected_500k_dollars
FROM sample_cost;
This Query is Non-Negotiable

This projection query must be run before any production AI pipeline. Paste your test query_ids and know your cost before you scale.

Shadow Waste Detection

Shadow waste is AI credit consumption that delivers little or no business value. AI services are serverless - no warehouse to watch, no query fails, no alert fires. Credits silently drain. The queries below detect the most common patterns.

8 Shadow Waste Patterns

Idle Search services, over-sized models, redundant calls, verbose prompts, agent loops, dev/test in prod, cost spikes, and missing attribution. The queries below target each pattern.

Query 5: Idle Cortex Search Detection

Shadow Waste #4

When to run: Weekly. Cortex Search services consume credits for serving (6.3 credits/GB/month) even with zero queries. Idle services = 100% waste.

SQL 
WITH costs AS (
    SELECT SERVICE_NAME,
           SUM(CASE WHEN CONSUMPTION_TYPE = 'SERVING'
                    THEN CREDITS ELSE 0 END)           AS serving_credits,
           SUM(CASE WHEN CONSUMPTION_TYPE = 'BATCH'
                    THEN CREDITS ELSE 0 END)           AS query_credits,
           SUM(CASE WHEN CONSUMPTION_TYPE = 'EMBED_TEXT_TOKENS'
                    THEN CREDITS ELSE 0 END)           AS embedding_credits
    FROM   SNOWFLAKE.ACCOUNT_USAGE.CORTEX_SEARCH_DAILY_USAGE_HISTORY
    WHERE  USAGE_DATE >= DATEADD('day', -30, CURRENT_TIMESTAMP())
    GROUP  BY SERVICE_NAME
)
SELECT SERVICE_NAME,
       serving_credits,
       embedding_credits,
       query_credits,
       ROUND(query_credits / NULLIF(serving_credits, 0) * 100, 1) AS query_pct
FROM   costs
WHERE  serving_credits > 0
ORDER  BY query_pct ASC;
Reading the Results

query_pct near 0% means the service is incurring standing serving cost but nobody is querying it. Consider suspending or dropping unused services:

ALTER CORTEX SEARCH SERVICE my_idle_svc SUSPEND SERVING;  -- stops standing cost
ALTER CORTEX SEARCH SERVICE my_idle_svc SUSPEND;          -- stops both serving + indexing
DROP CORTEX SEARCH SERVICE IF EXISTS my_abandoned_svc;    -- permanent removal

CONSUMPTION_TYPE values: SERVING (standing cost, 24/7), EMBED_TEXT_TOKENS (indexing on data change), BATCH (query usage).

Query 6: Over-Sized Model Detection

Shadow Waste #1

When to run: Monthly - developers default to the most capable (expensive) model. Compare credits_per_1M_tokens across models to find where a cheaper model would suffice.

Source: CORTEX_AISQL_USAGE_HISTORY - has TOKENS and TOKEN_CREDITS granularity.

SQL 
SELECT MODEL_NAME, FUNCTION_NAME,
       SUM(TOKENS)        AS total_tokens,
       SUM(TOKEN_CREDITS)  AS total_credits,
       ROUND(SUM(TOKEN_CREDITS)
             / NULLIF(SUM(TOKENS),0) * 1000000, 2)
                            AS credits_per_1m_tokens
FROM   SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AISQL_USAGE_HISTORY
WHERE  USAGE_TIME >= DATEADD('day', -30, CURRENT_TIMESTAMP())
GROUP  BY MODEL_NAME, FUNCTION_NAME
ORDER  BY credits_per_1m_tokens DESC;
Savings Potential: 40–75%

Replace AI_COMPLETE('sentiment...') with AI_SENTIMENT(text) for 40–70% savings. Replace AI_COMPLETE('classify...') with AI_CLASSIFY(text,[...]) for 50–75%. Task-specific functions are purpose-built and cheaper.

Query 7: Redundant / Duplicate AI Calls

Shadow Waste #2

When to run: Weekly - pipelines re-processing unchanged rows every scheduled run, multiple users running overlapping queries. 10K rows hourly = same tokens 24x/day.

Fix: Incremental processing (WHERE ai_processed_at IS NULL), cache AI results, tag pipelines with QUERY_TAG.

SQL 
-- Find repeated calls by user/day
SELECT h.USER_ID, u.NAME,
       h.FUNCTION_NAME,
       DATE_TRUNC('day', h.START_TIME)  AS day,
       COUNT(*)                         AS daily_calls,
       SUM(h.CREDITS)                   AS daily_credits
FROM   SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AI_FUNCTIONS_USAGE_HISTORY h
JOIN   SNOWFLAKE.ACCOUNT_USAGE.USERS u
       ON h.USER_ID = u.USER_ID
WHERE  h.START_TIME >= DATEADD('day', -7, CURRENT_TIMESTAMP())
GROUP  BY h.USER_ID, u.NAME, h.FUNCTION_NAME, day
HAVING COUNT(*) > 20
ORDER  BY daily_credits DESC;

Query 8: Verbose / Bloated Prompts

Shadow Waste #3

When to run: Monthly - prompts with excessive instructions or entire documents when only a paragraph is relevant. 50-word filler x 100K rows = 12.3 credits wasted.

Fix: Trim prompts, pre-filter with LEFT(document_text, 500), use AI_COUNT_TOKENS as pre-flight check.

SQL 
-- Compare avg tokens/call over time
SELECT FUNCTION_NAME,
       DATE_TRUNC('week', USAGE_TIME)  AS week,
       AVG(TOKENS)                     AS avg_tokens_per_call,
       SUM(TOKEN_CREDITS)              AS weekly_credits
FROM   SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AISQL_USAGE_HISTORY
WHERE  USAGE_TIME >= DATEADD('day', -60, CURRENT_TIMESTAMP())
GROUP  BY FUNCTION_NAME, week
ORDER  BY FUNCTION_NAME, week;
Prompt Engineering IS Cost Engineering

Rising avg_tokens_per_call over time = prompt bloat. Check TOKENS_GRANULAR for input vs output split. Truncating input can cut costs 50%+.

Query 9: Uncontrolled Agent Loops

Shadow Waste #5

When to run: Weekly - agents orchestrate multi-step tool calls. Without budgets, an agent can loop endlessly. Worst case: 100K+ tokens = 0.18+ credits per question.

Signal: MAX(TOKEN_CREDITS) >> AVG(TOKEN_CREDITS) = unbounded calls. Set token + time budgets on all agents.

SQL 
-- Find runaway agent calls
SELECT DATE_TRUNC('day', USAGE_TIME) AS day,
       MAX(TOKEN_CREDITS)            AS max_call,
       AVG(TOKEN_CREDITS)            AS avg_call,
       COUNT(*)                      AS total_calls
FROM   SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AISQL_USAGE_HISTORY
WHERE  USAGE_TIME >= DATEADD('day', -14, CURRENT_TIMESTAMP())
GROUP  BY day
ORDER  BY day DESC;

Query 10: Dev/Test in Production

Shadow Waste #6

When to run: Monthly - developers experimenting with AI using production credentials. Notebooks, ad-hoc queries, prototypes consuming prod credits without business value.

Fix: RBAC (REVOKE USE AI FUNCTIONS from dev roles), separate dev/prod accounts, enforce tagging.

SQL 
-- AI usage by role (LATERAL FLATTEN on ROLE_NAMES array)
SELECT r.VALUE::STRING   AS role_name,
       h.FUNCTION_NAME,
       COUNT(*)           AS calls,
       SUM(h.CREDITS)     AS total_credits
FROM   SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AI_FUNCTIONS_USAGE_HISTORY h,
       LATERAL FLATTEN(input => h.ROLE_NAMES) r
WHERE  h.START_TIME >= DATEADD('day', -30, CURRENT_TIMESTAMP())
GROUP  BY role_name, h.FUNCTION_NAME
ORDER  BY total_credits DESC;

Query 11: Week-over-Week Cost Anomaly Detection

Shadow Waste #7

When to run: Weekly - a code change or misconfigured schedule causes a spike. Unnoticed until the monthly bill. Example: WHERE clause change - 1K rows becomes 1M.

Fix: Activate account budget with email notifications, enable low-latency refresh (1hr vs 6.5hr default).

SQL 
-- Week-over-week anomaly detection
WITH daily AS (
    SELECT USAGE_DATE::DATE AS day,
           SUM(CREDITS_BILLED) AS credits
    FROM   SNOWFLAKE.ACCOUNT_USAGE.METERING_DAILY_HISTORY
    WHERE  SERVICE_TYPE IN ('AI_SERVICES','CORTEX_CODE_CLI','CORTEX_CODE_SNOWSIGHT')
      AND  USAGE_DATE >= DATEADD('day', -60, CURRENT_DATE())
    GROUP  BY day
)
SELECT day, credits,
       LAG(credits, 7) OVER (ORDER BY day)   AS last_week,
       ROUND((credits - LAG(credits, 7) OVER (ORDER BY day))
             / NULLIF(LAG(credits, 7) OVER (ORDER BY day), 0)
             * 100, 1)                         AS wow_pct
FROM   daily
ORDER  BY day DESC;
Reading the Results

wow_pct > 200 = spend more than tripled vs same day last week - investigate immediately. Even +50% week-over-week warrants review. Pair this with budget email alerts for proactive detection.

Query 12: Missing Cost Attribution

Shadow Waste #8

When to run: Monthly - without tagging, AI costs = one aggregated number. Nobody knows which team/project/pipeline owns the spend. Can't charge back, can't optimize.

Fix: Enforce ALTER SESSION SET QUERY_TAG='team=X,project=Y,env=prod'. Target: >90% credits should be tagged.

SQL 
-- Check tagging coverage
SELECT CASE WHEN QUERY_TAG IS NOT NULL AND QUERY_TAG != ''
            THEN 'Tagged' ELSE 'Untagged'
       END                              AS attribution,
       COUNT(*)                          AS calls,
       SUM(TOKEN_CREDITS)                AS credits,
       ROUND(SUM(TOKEN_CREDITS)
             / (SELECT SUM(TOKEN_CREDITS)
                FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AISQL_USAGE_HISTORY
                WHERE USAGE_TIME >= DATEADD('day', -30, CURRENT_TIMESTAMP()))
             * 100, 1)                   AS pct_of_total
FROM   SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AISQL_USAGE_HISTORY
WHERE  USAGE_TIME >= DATEADD('day', -30, CURRENT_TIMESTAMP())
GROUP  BY attribution;
Unattributed Spend Cannot Be Optimized

Query tagging is free - one SQL line per session. Use a consistent standard like team=<team>,project=<project>,env=<env> to enable automated cost reporting and chargeback.
